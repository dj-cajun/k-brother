### Identified Errors & Inconsistencies

1.  **Critical: Incomplete `NOTIFICATION_SPEC.md`**
    *   **Error**: The `01-Blueprints/NOTIFICATION_SPEC.md` file is severely truncated. It stops abruptly after the "네임스페이스:" header, providing no actual details about the Socket.IO events, data structures, or Discord Webhook payload. This is a crucial piece of the "realtime" and "notification" aspects of the system.
    *   **Impact**: Without this specification, the developers implementing the real-time and webhook features will lack necessary details, leading to potential inconsistencies or incomplete implementation.

2.  **Minor: Missing `default gen_random_uuid()` for PKs in `DB_SPEC.md`**
    *   **Inconsistency**: In the `DB_SPEC.md`, `users.id` explicitly has `default gen_random_uuid()`. However, `agents.id`, `coupons.id`, and `transactions.id` (all UUID PKs) are missing this `default` clause.
    *   **Recommendation**: For consistency and to leverage database-level UUID generation, add `default gen_random_uuid()` to `id` columns in `agents`, `coupons`, and `transactions` tables.

### General Feedback & Areas for Potential Improvement (Not Errors)

1.  **Clarity on `pg` direct vs. Prisma ORM**:
    *   The "기술 스택" section mentions "Prisma ORM 권장, 본 MVP는 pg 직접 사용". It might be helpful to briefly state *why* `pg` is being used directly for the MVP despite Prisma being recommended (e.g., "for minimal dependency footprint in MVP," "to fully control SQL for specific optimizations," or "to defer ORM learning curve post-MVP"). This is not an error, just an opportunity for added clarity.

2.  **Agent Authentication Mechanism**:
    *   The "API 명세 요약" lists `POST /api/agent/verify` as requiring "Agent" authentication. While the `agents` table has `login_id` and `password_hash`, and `middleware/auth.js` exists, the document doesn't explicitly state that agents also use JWT for authentication (which is a common pattern when users use JWT).
    *   **Recommendation**: Briefly clarify that agents also authenticate via JWT, or mention the specific authentication method for agents if it differs from users.

3.  **Error Handling Strategy**:
    *   While an MVP, a high-level note on error handling strategy (e.g., centralized error middleware, specific error codes, logging of errors) could be beneficial, even if not fully detailed.

4.  **Deployment/Infrastructure Considerations**:
    *   This is an architecture *specification*, but it's an MVP. For future versions, typical deployment targets (e.g., AWS, GCP, Vercel), scaling strategies, and monitoring tools would be valuable additions. For an MVP, this is usually out of scope, but good to keep in mind.

The document is of very high quality. The primary issue is the incompleteness of the `NOTIFICATION_SPEC.md`, which needs to be filled out with the actual details of the real-time and webhook notifications.
